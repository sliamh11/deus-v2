# ADR: Three-layer facade-prevention mechanism

**Status:** Accepted
**Date:** 2026-05-30
**Scope:** Layer 1 + docs (this PR): `.claude/wardens/code-review-rules.md`, `.claude/wardens/verification-rules.md`, `docs/CONTRIBUTING-AI.md`, this ADR. Layer 2 (planned, PR2): `.github/workflows/validate-pr.yml` job + `scripts/ci/flag_lint.py`. Layer 3 (planned, PR3 / fast-follow): `scripts/maintenance/orphan_sweep.py` + seed list (launchd generated via `--print-launchd`).

## Context

The LIA-133 audit found ~9 "built but not wired" components â€” code that was
implemented, unit-tested, and merged but never reachable in the runtime path.
Examples: the retrieval-param optimizer behind `DEUS_TREE_PARAMS` (never read
live until LIA-136), the multi-agent orchestrator behind `DEUS_MULTI_AGENT`
(still unwired, LIA-127), a write-only gcal cache nothing queried (deleted in
LIA-135).

The defining property: **every facade had passing unit tests.** Green unit
tests prove a unit behaves in isolation; they do not prove it is connected to
a live caller. The existing gates (`code-reviewer`, `ai-eng-warden`,
`plan-reviewer`, plus CI) all passed on every one of these merges. So the gap
is structural â€” no gate checked *reachability*, and no single per-diff gate
ever could catch "the consumer PR never landed weeks later."

## Decision

Three layers, each catching a different failure-time window. No single layer is
sufficient; the redundancy is intentional.

### Layer 1 â€” connectivity rule (commit-time, judgment-based)

A `connectivity-wiring` rule in `code-review-rules.md` (severity **warning**).
Fires when a diff adds a new `DEUS_*` flag gate, a new integration/entry-point
module, or a new top-level runtime module. The author must show the
producerâ†’consumer edge or cite a deferred-wire Linear issue.

Severity is `warning`, not `blocking`, deliberately: it is an LLM judgment call,
and a hard block on a judgment call is how gates earn their bypass. The
deterministic block is Layer 2. A short claim-time advisory companion
(`wire-reachability`) lives in `verification-rules.md` â€” note that the
`verification-gate` warden has no `tools` binding in `config.json` and is
therefore advisory only, so the commit-enforced copy must live in
`code-review-rules.md`. The two entries are cross-referenced, not duplicated,
to respect the "never duplicate content across files" core rule.

### Layer 2 â€” flag-lint (merge-time, deterministic)

A CI job scans the PR diff for **net-new** `DEUS_*` flag gates â€” a flag name
absent from the base ref. Each must cite a Linear issue (`LIA-NNN`/`#NNN`) in
an adjacent comment, or carry a `dev-only` escape, or CI fails. Net-new scoping
is essential: ~40+ `DEUS_*` flags already exist, so a check that fired on every
flag *read* would flood. The cite/escape is accepted in `#`, `//`, and `/* */`
comment syntaxes (~60 flags live in `.py`/`.sh` files). The linter excludes its
own source (`scripts/ci/`) and test fixtures (`scripts/tests/`) from the scan so
it does not fire on itself.

This is the load-bearing layer â€” deterministic and unbypassable in review. True
unbypassability additionally requires adding the job to branch-protection
required checks, which is a one-time repo-admin action outside this change.

### Layer 3 â€” hybrid codegraph+grep orphan-sweep (drift-time, local)

A periodic job reports exported symbols (function/class/method) in a curated set
of integration modules that look like facades, using **dual-evidence
corroboration**: a symbol is flagged only when BOTH (a) the codegraph index shows
no caller from a non-test file other than its own, AND (b) `git grep` finds the
name in no non-test file outside its own definition. This catches the LIA-137
pattern: merged behind a flag, tracked, but the wire never came.

Why hybrid, not codegraph-only: investigation during this work found codegraph's
caller graph has **silent edge gaps** â€” e.g. `createMessageOrchestrator` is called
at `src/index.ts:339` yet codegraph records zero callers (the call is absent from
both `edges` and `unresolved_refs`, even after a clean reindex), and method-dispatch
calls like `orchestrator.startMessageLoop()` are unresolvable without type inference.
A codegraph-only zero-caller sweep would therefore false-positive on live entry
points â€” the same flood the rejected blanket check warns about. The grep arm
suppresses those false positives; the AND-logic must not be relaxed to OR.

**This layer is macOS-local only and does NOT run on Linux or in CI.** It reads
the gitignored `.codegraph/codegraph.db` produced by the local codegraph daemon
and is scheduled via launchd (generated by the tool's `--print-launchd`). This
diverges from the original handoff sketch ("weekly GitHub Actions job") because
the codegraph index is a host-local daemon artifact, not a portable CI input. It
is a host-side drift sweep, not a
portable gate. Curated (opt-in by module), not blanket â€” see below.

## What this is NOT â€” rejected blanket zero-caller check

A naive "flag any exported symbol with zero callers across the whole repo" check
was rejected. It floods false positives:

- **Flag-gated code is reachable** â€” the flag just isn't on yet. A zero-caller
  check cannot distinguish "facade" from "feature shipped dark on purpose".
- **MCP tools and channel handlers are entry points** reached by dispatch /
  registration, not by a static caller edge â€” they look callerless to a graph.
- **CLI subcommands** are invoked by argv, not by a code edge.

All three layers are therefore deliberately *scoped*: Layer 1 to new
flags/modules with a judgment check, Layer 2 to net-new flags with a citation
escape, Layer 3 to a curated module list. We trade completeness for a usable
false-positive rate â€” a noisy gate that everyone learns to ignore prevents
nothing.

## Consequences

- New `DEUS_*` flags now carry a lightweight obligation: cite the tracking issue
  or mark dev-only. This is cheap and creates a durable paper trail from flag to
  intent.
- The mechanism is itself subject to the facade trap (its tests will be green).
  The real proof, recorded in the PR2 body when Layer 2 ships, will be a manual
  red-green run of `flag_lint.py` against the historical `DEUS_MULTI_AGENT`
  introduction commit (its parent as base ref) â€” not the hermetic fixtures.
- Layer 3 will not protect non-macOS contributors or CI; that limitation is
  documented loudly here and in `CONTRIBUTING-AI.md` rather than papered over.

## Reversibility

Each layer is independently revertible. Layer 1 is a rules-file edit, Layer 2 is
a CI job + script, Layer 3 is a local script + launchd unit. Removing any one
degrades coverage of its specific failure window but breaks nothing.
