# Deus v2.0.0

Deus v2.0.0 publishes the Deus V2 runtime architecture and its supported
migration state. It does not change the default backend: Claude remains the
compatibility baseline and default, while `deus-native` is available by
explicit opt-in. The default flip tracked by H2/LIA-434 remains blocked by
H1/LIA-433's reliability evidence gate.

## What ships in V2

- A backend-neutral `AgentRuntime` registry and resolver with task, group,
  and global backend selection. Sessions are backend-scoped and are never
  resumed across a backend mismatch.
- `deus-native`, a host-side agent loop built with LangChain JS `createAgent`
  and LangGraph checkpointing. It is registered in the production runtime and
  can be selected with `DEUS_AGENT_BACKEND=deus-native` or a group/task
  override.
- Deus-owned middleware for lifecycle context, control-group memory
  retrieval, checkpoint-aware compaction, and declarative permission
  profiles. Telemetry middleware builds call metadata (provider, model id,
  message count) as part of the middleware stack's per-layer log output, but
  production `runTurn()` currently discards that log (only the middleware
  itself is retained) — it is not wired to any consumer today, and this is
  metadata only regardless, not real token/usage accounting. Real
  per-message usage accounting DOES exist in
  production, via a separate mechanism (`TurnUsageCollector`, exposed as
  `RunResult.usage` and per-message `usage` events). The warden-interception
  mechanism (`wrapToolCall`) is real and production-wired but its trigger is
  currently dormant — the broker allowlist restricts `deus-native` to
  `web_search`/`web_fetch`, so no mutating tool call exists yet to intercept.
  Transcript persistence (permission-restricted, session-keyed JSONL) is a
  separate mechanism from the middleware stack, not part of it.
- Persistent `deus-native` sessions, validated main/per-role model selection,
  the `deus chat` terminal client with `/plan on|off`, and one-shot nested
  subagent dispatch (`dispatch_nested_agent`) with independently validated
  output contracts.
- Context loading for the standard instruction files (`AGENTS.md`,
  `CLAUDE.md`, `AI_AGENT_GUIDELINES.md`) across applicable group, global,
  project, vault, and additional roots — this covers those specific files,
  not general content-parity with every context surface v1 could load.
- Container protocol portability for direct, non-default `deus-native` turns
  and lazy `SKILL.md` instruction-pack loading. This is a portability proof,
  not a production container routing change.
- A turn-lifecycle activity event stream (G5/LIA-432): an authenticated SSE
  fan-out of runtime lifecycle events. Parent-level events (tool-call, usage,
  answer, completion) are emitted once the turn's buffered `agent.invoke()`
  completes; nested-dispatch usage is recorded as each nested call finishes
  during the turn. Either way, this is not live/real-time tool-call or phase
  progress — the turn's actual answer and tool results remain fully buffered,
  distinct from model response streaming.
- Linear pipeline backend selection with a capability-readiness guard.
  `deus-native` Linear coding work is refused and moved to Manual Review
  Required instead of being reported as a false success while mutation and
  commit tools remain unavailable.

## Configuration and rollback

Existing installations stay on Claude unless explicitly configured
otherwise. To try Deus-native globally, set `DEUS_AGENT_BACKEND=deus-native`
in the project-root `.env` and restart Deus — macOS via `deus build`
(auto-restarts), Linux via `deus build` followed by
`systemctl --user restart deus`. Windows has no credential-neutral restart
command today: `deus auth` is the only rebuild/restart option and it requires
a Claude OAuth token, so it fails for installations using the
`ANTHROPIC_API_KEY` credential-proxy mode — those users should run
`npm run build` and restart the Deus service manually instead (a
credential-neutral `deus-cmd.ps1` command is a tracked follow-up). See
[Using different AI backends](../MULTI_BACKEND.md#try-deus-native) for the
full OS-specific instructions. Verify the selected
backend with `/context`. A group can opt in with
`/settings backend=deus-native`, and a scheduled task can carry an
`agent_backend` override — though the scheduled-task path remains an H1
release blocker and should be treated as unverified.

To roll back, set `DEUS_AGENT_BACKEND=claude` (or remove the global
override), restart Deus the same OS-appropriate way, and verify `/context`
reports Claude. Clear a group override with `/settings backend=default`, or
force Claude for that group with `/settings backend=claude`.

Backend switches do not migrate sessions between runtimes. Each backend keeps
its own session reference, so returning to Claude resumes or starts a Claude
session rather than handing a Deus-native checkpoint to the Claude adapter.

## Compatibility and known limitations

- Claude remains the default. H2/LIA-434 will not flip the default until
  H1's tool-loop reliability evidence records a GO.
- The host-side Deus-native runtime currently exposes `web_search`,
  allowlisted `web_fetch`, and `dispatch_nested_agent`; shell, filesystem,
  multimodal, and streaming capabilities remain unavailable.
- Non-control groups do not yet receive per-turn personal-memory retrieval
  because the current retriever is not group-scoped (`AAG-014`).
- Public-ingress groups are refused on Deus-native to avoid weakening the
  existing container security profile.
- Deus-native Linear coding dispatch remains capability-blocked until an
  isolated mutation/commit tool surface and the replay-safety idempotency
  contract land (`AAG-017`).
- End-to-end scheduled-task selection and container tool-broker coverage
  remain H1 release blockers.
- Claude Code, Codex CLI, and OpenCode are optional Tier 2 development
  clients. They do not select, replace, or provide guarantees for the Tier 1
  product runtime.
